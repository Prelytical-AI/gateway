Prelytical Secure SQL Gateway
Architecture Options

Three ways to deploy AI-assisted, read-only SQL analytics in sensitive environments — with trade-offs for speed, simplicity, and data residency.

Every option keeps the same application security model: read-only SQL login, SELECT-only guardrails, curated schema access, local audit logging, and a browser UI bound to localhost:8080 on the gateway VM. What changes is where the LLM runs and how much infrastructure the client operates.

Three deployment options

Choose based on POC vs production, available GPU hardware, and how strictly data must stay inside the client boundary.

Option 1

Same-VM (All-in-One)

SQL Server, gateway, and Ollama on one Windows VM

┌─────────────────────────────────────┐ │ Single Windows VM │ │ │ │ ┌──────────┐ ┌──────────┐ │ │ │ SQL │ │ Gateway │ │ │ │ Server │◄─┤ :8080 │ │ │ └──────────┘ └────┬─────┘ │ │ │ localhost │ │ ┌──────▼──────┐ │ │ │ Ollama CPU │ │ │ │ :11434 │ │ │ └─────────────┘ │ └─────────────────────────────────────┘ Best for: POC, demos, small datasets

Pros

  • Simplest install — one VM, one runbook, no networking between boxes
  • Lowest client IT overhead; matches existing SQL Server VM patterns
  • All data stays on one machine; no cross-VM traffic to secure
  • Works offline after initial model download
  • Ideal for proof-of-concept and client demos

Cons

  • CPU-only inference is slow (often 2–5+ minutes per question)
  • Ollama competes with SQL Server for RAM and CPU
  • Two LLM calls per question (SQL generation + summary) doubles latency
  • Not suitable for interactive production use at scale
  • Large models may not fit comfortably alongside SQL workloads
Easiest setup Slow responses Full data residency
Option 2

Split-VM (SQL + GPU Inference)

Gateway on SQL VM; Ollama on a dedicated GPU box in the same VPC/LAN

Recommended for production
┌──────────────────────┐ private ┌──────────────────────┐ │ VM 1: SQL + Gateway │ ── :11434 ─► │ VM 2: GPU Inference │ │ │ HTTP only │ │ │ SQL Server │ │ Linux + NVIDIA │ │ Gateway :8080 │ ◄─────────── │ Ollama :11434 │ │ MODEL_BASE_URL ─────┼── 10.x.x.x │ (no SQL, no UI) │ └──────────────────────┘ └──────────────────────┘ Same VPC / on-prem network — no public internet

Pros

  • Fast inference — responses in seconds instead of minutes
  • SQL Server and LLM workloads are isolated; no resource contention
  • Data never leaves the client environment (private IP only)
  • Gateway change is minimal: one MODEL_BASE_URL in .env
  • GPU VM can be sized independently; scale or upgrade without touching SQL
  • Security groups restrict port 11434 to the SQL VM only

Cons

  • Two VMs to provision, patch, and monitor
  • Requires GPU quota / hardware (e.g. AWS g4dn.xlarge)
  • Client IT must configure VPC networking and security groups
  • Questions, schema metadata, and query results traverse the private network to the GPU VM
  • Slightly more complex bootstrap (two install scripts)
Fast inference Two VMs Client-owned boundary
Option 3

Cloud-Hosted Inference

Gateway on client VM; LLM calls a hosted API (Prelytical cloud or third party)

Not recommended for sensitive data
┌──────────────────────┐ ┌──────────────────────┐ │ Client VM │ HTTPS │ External Cloud │ │ │ ─────────────► │ │ │ SQL Server │ questions, │ Hosted LLM API │ │ Gateway :8080 │ schema, rows │ (Prelytical / etc.) │ └──────────────────────┘ └──────────────────────┘ Data crosses client network boundary

Pros

  • No GPU hardware or local model management
  • Fastest time-to-value if cloud API is already approved
  • Always on latest models without client-side upgrades
  • Elastic capacity — no sizing a GPU VM
  • Single VM on client side (SQL + gateway only)

Cons

  • Question text, schema metadata, and query result rows leave the client network
  • Conflicts with typical healthcare, finance, and government data policies
  • Requires outbound internet and third-party DPAs / BAAs
  • Not aligned with Prelytical Gateway’s on-prem security story
  • Latency and availability depend on external service SLA
  • Audit trail must account for off-box data processing
No GPU needed Data leaves client Policy risk

Which option should we choose?

A practical decision path based on what we’ve seen in client deployments and POC testing.

1 Start here: Option 1

Use Same-VM for the first install, demo, or security review. Prove read-only SQL, guardrails, and audit logging without asking the client for GPU infrastructure. Expect slow Ask responses — set that expectation upfront.

2 Move here for production: Option 2

Once the POC is approved, deploy a GPU inference VM in the same VPC. Run install/bootstrap-inference-linux.sh on that box (see GPU_INFERENCE_VM.md). Update MODEL_BASE_URL=http://<private-ip>:11434/v1 on the SQL VM and skip local Ollama.

3 Avoid for sensitive data: Option 3

Cloud inference is only appropriate when the client explicitly allows row-level data to leave their network and has contractual coverage with the API provider. Prelytical Gateway’s on-prem product is designed for Options 1 and 2.
