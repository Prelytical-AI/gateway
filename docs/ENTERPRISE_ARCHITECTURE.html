Enterprise Target Architecture

Hybrid Prelytical
Azure Product + Client GPU Inference

The production-ready shape for regulated clients: Prelytical hosts the executive experience in Azure; the warehouse, SQL execution, and AI inference stay on the client network.

One sentence: Prelytical runs in Azure for the product; your data and AI stay on your network on a private GPU — fast, governed, and not sent to OpenAI.

Enterprise rollout comes down to two gating decisions: provision GPU inference on client infrastructure, and align with security on what warehouse context may reach the model. Everything else — curated views, read-only SQL, audit, patching — is implementation inside that boundary.

Target topology

Two trust zones. The cloud zone is the product shell. The client zone is where sensitive data and inference live. HTTPS connects them; raw warehouse rows do not leave the client zone unless explicitly approved.

Azure — product layer
  • Executive UI and Brief Studio
  • Workspaces, users, domains, governance settings
  • Brief run history and export delivery
  • SaaS updates without touching client SQL
  • Optional cloud models for approved non-PII flows only
Client — data & inference
  • SQL Server (source of truth)
  • Read-only SQL execution and guardrails
  • GPU VM with Ollama (private network only)
  • Schema-first brief generation and Ask
  • Local audit log for SQL and model events
Connector: Platform API calls the on-prem processing agent over HTTPS (outbound from client, or site-to-site VPN / private link). Agent never exposes SQL credentials to Azure.

What this is not: Putting the GPU in Prelytical’s public cloud and pulling SQL over the internet. That rarely passes security review. The GPU belongs in the client’s network, reachable from the processing agent — same pattern as Option 2 today.

Two gating decisions

Everything else is execution once these are settled with client IT and security.

1. Inference path — GPU on client network

For production executive use, CPU-only inference is too slow. Plan a private GPU VM:

  • AWS g4dn.xlarge or Azure NC-series (~$400/mo on-demand)
  • Ollama on private IP; port 11434 restricted to agent VM
  • Models: qwen2.5-coder:7b default; 14b+ for quality on same GPU tier

2. Privacy boundary — what may reach the model

Security and legal must sign off on prompt content in the client zone:

  • Schema metadata (table/column names)
  • Query result rows for Ask summarization (capped)
  • Brief context derived from live warehouse shape
  • What may sync back to Azure (scores, HTML exports vs raw schema)

Is Ollama good enough?

Yes for on-prem enterprise — with eyes open about the quality ceiling vs cloud GPT.

Workload Ollama (7B–14B on GPU) GPT-class (platform cloud)
Guarded SQL Ask on ai views Good enough Better on messy schemas
Schema-first executive brief Solid with server-side renderer Richer nuance and prose
Brief + deep business context Thinner without rich input Strongest
Data privacy Best — stays on client Requires explicit approval

Quality dial without leaving the client: larger Ollama models on the same GPU (qwen2.5-coder:14b), richer ai views, and business context in the brief form. The gap vs GPT is real but manageable for schema-first planning — not row-level magic.

When the client asks about Copilot / Azure OpenAI

M365 Copilot is not a plug-in API for Prelytical. Azure OpenAI is the realistic cloud alternative — with a different privacy trade.

Copilot clarification: “We have Copilot in our Azure tenant” usually means M365 Copilot and/or Azure OpenAI. Only Azure OpenAI can power a custom backend — and it sends schema and query context off the SQL VM into Azure. That is an explicit policy choice, not the default hybrid design.

Enterprise readiness checklist

For client IT, security, and Prelytical deployment planning.

  • Security approves schema/metadata in on-prem model prompts
  • GPU VM provisioned in client VPC with private IP only
  • Firewall: 11434 open from agent VM → GPU VM only
  • Read-only SQL login on curated ai views
  • Processing agent deployed (gateway or headless v2)
  • Platform tenant wired to agent endpoint (HTTPS / VPN)
  • Ollama model pulled and warmed (qwen2.5-coder:7b+)
  • Audit retention and VM patching ownership defined
  • Agreement on what brief artifacts sync to Azure
  • POC completed on Option 1 or 2 before hybrid cutover
